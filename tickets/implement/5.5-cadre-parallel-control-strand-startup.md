## Goal

After the lazy-strand-bring-up ticket, the LogHistory cold path still serializes:

```
control libp2p start â†’ control DB init â†’ strand libp2p start â†’ strand DB init
```

But `StrandInstanceManager.startStrand` (which is what `CadreNode.addStrand` calls into) only depends on inputs available at construction time â€” `config.privateKey`, `config.storage`, `config.network`, `config.profile`, the strand row, the sApp config. It does NOT depend on the control libp2p node, the control DB, the strand watcher, or any control-network state. The `if (!this.running)` guard at the top of `CadreNode.addStrand` is purely defensive; the underlying work is structurally independent.

If we relax that guard (or expose a "register strand alongside start" config), mobile can do:

```ts
const controlPromise = node.start();
const strandPromise = node.addStrand(...); // no longer waits for running===true
await Promise.all([controlPromise, strandPromise]);
```

LogHistory cold start drops from `control + strand` to `max(control, strand)` â€” close to a 50% improvement when the two stages are similar in duration.

## Risks to verify before changing

- `StrandWatcher` polls the control DB every 5s and triggers `handleStrandAdded` for any matching row. If the watcher fires for our strand id while `addStrand` is still running, both paths funnel into `strandManager.startStrand`, which short-circuits when the strand is already in its instance map. Verify the short-circuit is race-safe under the new ordering (it inserts into `instances` *before* the heavy work).
- `HibernationManager.start()` is called inside `CadreNode.start()`; `hibernationManager.trackStrand(instance)` is called from `addStrand`. If `trackStrand` requires the manager to be started, the parallel path will fail. Audit `hibernation-manager.ts` and either start the manager in the constructor or queue `trackStrand` calls until `start()` runs.
- `CadreNode.emit('strand:started', ...)` may now fire before `emit('control:connected', ...)`. Document this ordering or sequence the emits explicitly inside `start()`.
- `CadreNode.cleanup()` runs on `start()` failure and may try to `stopStrand` strands that are mid-startup. Make sure cleanup is idempotent across "strand started in parallel" cases.

## Approach

In cadre-core (`C:/projects/sereus/packages/cadre-core`):

- Drop the `if (!this.running)` check in `CadreNode.addStrand`. Replace with a check that the constructor has run (i.e. `strandManager` is initialized â€” that's already always true).
- Audit `HibernationManager`: if `trackStrand` is unsafe before `start()`, either move the start into the constructor (cheap; just initializes timers) or buffer pre-start `trackStrand` calls and flush during `start()`.
- Document the new ordering rule in the `addStrand` JSDoc: "May be called any time after construction; does not require `start()` to have completed."
- Add a unit test: `addStrand` + `start()` invoked concurrently â†’ both promises resolve, strand and control both end up running, no spurious errors.

In mobile (`apps/mobile/src/services/CadreService.ts`):

- Inside `ensureStrandReady` (introduced by the lazy-bring-up ticket), kick off `node.addStrand(...)` in parallel with `node.start()` instead of awaiting start first.
- Keep `ensureControlReady` returning when `node.start()` resolves (so SereusConnections still gets early control readiness without waiting on the strand).
- Re-emit timings: `cadreNode.start (control libp2p + control DB)`, `addStrand (strand libp2p + strand DB + schema apply)`, plus a new `parallel max` so the after-numbers compare cleanly to the lazy-bring-up baseline.
